1. System Overview & Core Capabilities

The Ethred Backend is a multi-tenant, role-based real estate platform designed specifically for the Ethiopian market. It handles property discovery, user identity, agency delegation, buyer-seller communication, Ethiopian Birr (ETB) payments via Chapa, real-time messaging, and admin content moderation.

Key Architectural Characteristics

  • Authentication: Stateless JWT stored in HTTP-Only cookies with sliding refresh tokens, email OTP verification, and Google OAuth 2.0.
  • Authorization: Granular Role-Based Access Control (RBAC) supporting 5 distinct user roles.
  • Spatial Search: PostgreSQL + PostGIS integration for coordinate-based geographic proximity queries (latitude/longitude).
  • Media Optimization: Automated multi-resolution WebP image transcoding (400px, 800px, 1200px) via Sharp.
  • Payment Verification: Cryptographic HMAC-SHA256 signature verification for Chapa webhooks with automated state transitions.
  • Auditing: Comprehensive audit logging capturing actor, action, target entity, and diffs for every state-changing operation.

2. User Roles & Privilege Matrix

The system enforces 5 authenticated roles plus an unauthenticated Public role:

Role Code Display Name Target Persona Access Summary
PUBLIC Guest Visitor Anyone Search properties, view active listings, initiate registration/login.
BUYER Property Buyer / Tenant Home seekers Search, save favorites, send inquiries, report listings, manage own profile.
SELLER Independent Seller / Landlord Property owners All Buyer privileges + create listings, upload media, initiate promotion payments, manage own inbox.
AGENCY_ADMIN Agency Owner / Manager Real estate firms All Seller privileges + create agency profile, invite agents, assign listings, view agency analytics.
AGENCY_AGENT Agency Agent / Broker Employed brokers All Seller privileges on behalf of their assigned agency.
ADMIN Platform Administrator Ethred staff Full system control: listing moderation queue, user management/banning, agency verification, audit log inspection, platform analytics.

4. Key Business Logic Processes & Workflows

4.1. Authentication & Session Flow

User (Email/Pass) ──► Auth Service ──► Verify Hash ──► Sign Access Token (15m) & Refresh Token (7d) │ Set HTTP-Only Cookie ("ethred_token") │ Subsequent Requests: Passport JWT Strategy Extracts cookie/Bearer -> Attaches req.user

4.2. Property Moderation State Machine

[SELLER Creates] ──► DRAFT │ ▼ (Seller Submits) PENDING ───(Admin Rejects)───► DRAFT (feedback sent) │ ├────(Admin Approves)─────► APPROVED (live in search) │ │ └────(3+ Scam Reports)─────────┴──► SUSPENDED

4.3. Chapa Payment & Promotion Workflow

Seller ──► POST /payments/initiate ──► Create PENDING Invoice ──► Request Chapa Checkout URL │ Seller Pays on Chapa │ Chapa Webhook Callback ──► POST /payments/chapa-webhook ◄──────────────────┘ │ ├─► 1. Verify HMAC-SHA256 Signature (x-chapa-signature) ├─► 2. Call Chapa API to verify tx_ref (anti-replay check) ├─► 3. Transition BillingInvoice status: PENDING ──► COMPLETED ├─► 4. Activate Property Promotion: is_featured = true (30 days) └─► 5. Send Email Receipt to Seller

4.4. Automated Scam Suspensions

When a user reports a listing via POST /inquiries/report/:propertyId:

  1. System logs an AuditLog entry with action LISTING_REPORTED.
  2. System counts unique report audit logs for that property.
  3. If total reports ≥ 3 and listing status is APPROVED:
    • Listing status is automatically updated to SUSPENDED.
    • System logs LISTING_AUTO_SUSPENDED audit entry.
